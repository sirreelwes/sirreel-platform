### Commit 2: Wire Contract Review AI to read and enforce playbook

**Files to modify (find by inspection):**
- AI prompt construction file in `src/lib/contracts/` (likely the file that builds the system prompt for contract review)
- API route handling contract review requests (likely `src/app/api/tools/contract-review/[id]/route.ts` or similar)
- Contract review UI page (likely `src/app/(routes)/tools/contract-review/[id]/page.tsx` or similar)

**Behavior changes:**

1. **Load playbook at review time.** Read `contract-negotiation-playbook.md` file contents on each contract review request (or load once at server start, depending on architecture preference — file size is small enough that per-request read is fine).

2. **Inject into system prompt.** Add playbook content under a heading `# SirReel Negotiation Positions` in the Claude system prompt for every contract review.

3. **Update prompt instructions** to direct the AI to:
   - Use "Preferred" language for initial counters
   - Surface "Acceptable Fallback" only when the operator marks the clause as second-round negotiation
   - Reject anything in "Do Not Accept" lists
   - **CRITICAL:** Enforce Non-Negotiable Hard Limits absolutely — especially §1 (third-party-only indemnity is never accepted, never offered as fallback)
   - Flag any draft counter missing "Hard Musts" for human review (return a flag in the response payload)

4. **Add second-round toggle UI.** On the contract review page, add a per-clause toggle: "This is a second-round negotiation." When toggled on, the AI considers Fallback language for that clause. When off (default), AI uses Preferred only.

5. **Add operator-flag rendering.** When AI flags a clause for operator review (e.g., because client redline violates a Hard Must or the AI can't generate a counter that satisfies all constraints), surface this prominently on the review UI with a "⚠️ Needs operator review" badge.

### Commit 3: Fix ContractDocument.tsx rendering bug

**Bug description:**

The counter-PDF generated by the Contract Review feature has two rendering issues:

1. **Clause 22–26 misalignment.** Counter PDF for review `RA_2026_PM_rdl-counter.pdf` (generated 2026-05-11) shows:
   - Clauses 22 and 23 missing from review section entirely
   - Clause 24 header "Entire Agreement ACCEPTED" — body shows Return clause content
   - Clause 25 header "Applicable Law COUNTERED" — body shows Additional Equipment content
   - Clause 26 header "Arbitration COUNTERED" — body shows Entire Agreement content
   - Pattern: content shifted by 2 positions relative to headers

2. **Accept decisions render AI commentary instead of accepted language** (per existing bug tracker notes). When operator marks a clause "Accept," the counter PDF renders the AI's commentary/reasoning text instead of `aiResponse.changes[i].proposed` (the actual accepted clause language).

**Hypothesis:** Both bugs likely stem from index mapping between canonical clause numbers and `aiResponse.changes[]` array positions in `src/lib/contracts/ContractDocument.tsx`. Possibly a 1-indexed vs 0-indexed mismatch, or an off-by-N issue when certain clauses are skipped (e.g., the system may skip clauses with no client redline, causing position drift).

**Files to investigate:**
- `src/lib/contracts/ContractDocument.tsx` (rendering logic for counter PDF)
- Wherever `aiResponse.changes` is constructed (likely the same `src/lib/contracts/` directory)
- The schema mapping canonical clause numbers to AI response indices

**Test plan:**
- Regenerate the counter PDF for the same review that produced `RA_2026_PM_rdl-counter.pdf`
- Verify all 29 clauses render with matching headers and bodies
- Test Accept decision: mark a clause "Accept," regenerate PDF, verify the rendered body is the accepted clause language (not commentary)
- Test Counter decision: regenerate, verify rendered body is the AI's suggested counter language

**Commit message:** `fix(contracts): correct clause indexing in counter PDF rendering`
